극도로 발전된 명세(프롬프트)는 코드와 구별되지 않는다

Karpathy가 "가장 핫한 새 프로그래밍 언어는 영어"라고 했을 때 많은 사람이 고개를 끄덕였습니다. 저도 그랬습니다. 그런데 직접 자연어로 에이전트를 제어하는 문서를 쓰고, 그게 깨지는 걸 반복해서 겪고 나니 생각이 조금 달라졌습니다. 영어가 프로그래밍 언어가 된 게 아니라, 프로그래밍 언어처럼 써야만 동작하는 게 아닐까 싶습니다.

배경

Claude Code에는 SKILL이라는 시스템이 있다. .claude/skills/ 디렉토리에 마크다운 문서를 두면 AI가 그걸 읽고 특정 작업을 수행한다. Slack에 메시지를 보내거나, DB 변경을 승인하거나, 코드를 검증하거나.

한동안 이걸 만들고 부수는 과정을 반복했습니다. 문제는 같은 문서가 어떤 날은 되고 어떤 날은 안 된다는 점이었습니다. 대화 맥락이 달라지면 AI가 같은 문장을 다르게 해석했습니다. 그래서 SKILL 문서의 품질을 검증하는 프레임워크를 만들었습니다. 이 과정에서 느낀 것들을 정리했습니다.

코드는 허용 목록이고, 프롬프트는 차단 목록이다

DB 변경을 승인하는 SKILL을 만들었습니다. 처음 버전은 이랬습니다.

## 프로세스
1. SQL 검증을 수행한다
2. Jira 티켓을 확인한다
3. 승인 처리를 한다

대부분의 경우 잘 작동했습니다. 그런데 가끔 AI가 SQL 검증 단계에서 "이 쿼리 성능 이슈가 있어서 수정해드리겠습니다"라며 쿼리를 고쳐버렸습니다. 시킨 적 없는 일이었습니다. 그런데 "고치지 마라"고 쓴 적도 없었습니다.

이게 코드와의 결정적 차이입니다. validateSQL(query)라는 함수를 짜면, 그 함수는 validation 로직 안에 있는 것만 수행합니다. 쿼리를 수정하는 코드를 넣지 않았으면 수정은 애초에 불가능합니다. 당연한 얘기입니다. 그런데 자연어로 "SQL 검증을 수행한다"라고 쓰면, AI는 "검증" 안에 수정도 포함될 수 있다고 판단합니다.

코드는 작성한 것만 수행합니다. 작성하지 않은 동작은 애초에 불가능합니다. 반면 자연어로 쓴 지시는, 명시적으로 막지 않으면 AI가 알아서 해버립니다. 통제의 기본값이 정반대인 셈입니다.

방화벽에 비유하면, 코드는 화이트리스트 방화벽에 가깝습니다. 허용 규칙에 있는 트래픽만 통과시킵니다. 프롬프트는 그 반대로, 블랙리스트 방화벽처럼 차단 규칙에 없는 트래픽은 전부 통과시킵니다.

실제로 어떻게 닫아갔는가

위의 SQL 수정 문제를 고치자 또 다른 구멍이 나왔습니다.

금지 사항을 추가했더니 이번엔 Slack 링크가 아닌 입력이 들어왔을 때 AI가 알아서 URL을 추측해서 처리했습니다. 그걸 막았더니 이번엔 Jira 티켓이 이미 승인된 상태인데도 또 승인 요청을 보냈습니다. 한 구멍을 막으면 다른 구멍이 터졌습니다. 블랙리스트 방화벽의 숙명대로, 막지 않은 곳은 뚫렸습니다.

검증 프레임워크를 만든 건 이 시행착오를 체계화하고 싶어서였습니다. SKILL이 깨지는 패턴을 모았더니 몇 가지로 분류가 됐습니다.

먼저, AI가 지시 범위를 넘어서 자기 판단으로 행동하는 문제가 있었습니다(Intent Drift). as needed, if appropriate, consider 같은 표현이 있으면 거의 확실히 터졌습니다. 이런 위임 표현은 코드로 치면 타입이 any인 변수와 비슷합니다. 뭐든 들어갈 수 있으니 뭐든 하게 됩니다. 그래서 체크리스트를 만들었습니다. 금지 항목이 최소 2개 이상 명시돼 있는지, 위임 표현이 포함돼 있지 않은지, AI가 재량으로 분기할 수 있는 지점에 명시적 기준이 있는지를 확인합니다.

대화 맥락에 오염되는 문제도 있었습니다(Context Contamination). 직전에 "공격적으로 리팩토링해"라는 대화가 있었으면, 바로 다음에 호출한 SKILL도 공격적으로 동작했습니다. 함수로 치면 말이 안 되는 일입니다. calculateDiscount(price, rate)를 호출하는데 직전에 호출한 함수가 뭐였는지에 따라 결과가 달라진다면, 전역 변수를 쓴 적도 없는데 이상한 일입니다. 그런데 LLM에서는 이게 일상적으로 벌어집니다. 전체 대화가 하나의 거대한 전역 상태인 셈입니다.

이 문제는 격리 선언("이전 맥락을 무시하라")으로는 못 막는다는 걸 금방 알게 됐습니다. 그래서 방향을 바꿨습니다. 오염 자체를 막는 대신, 오염됐을 때 피해 범위를 측정하는 쪽으로 옮겼습니다. 읽기 전용 SKILL이면 오염돼도 큰 문제가 아닙니다. 하지만 DB를 바꾸거나 Jira를 승인하는 SKILL이 오염되면 프로덕션에 영향이 갑니다. 그래서 blast radius를 Local/Team/Production 3단계로 나누고, Production 단계에서 사전 확인 게이트가 없으면 FAIL 판정을 내리도록 했습니다.

이런 식으로 총 7개 차원이 만들어졌습니다. SKILL 하나를 검증하면 이런 결과가 나옵니다.

/validate-skill approve-db-changes

[1] Intent Drift Prevention       WARN — 금지 섹션 없음. 금지 항목 0개
[2] Intent Corruption Prevention  FAIL — 불변 조건 섹션 없음. 입출력 예시 없음
[3] Intent Completeness           FAIL — 미정의: 비정상 입력, 누락된 티켓, API 실패
[4] Trigger Clarity               PASS — 트리거 3개 명확. 충돌 없음
[5] Environment Dependencies      WARN — 토큰 명시됐으나 획득 경로 미기술
[6] Context Contamination Risk    WARN — Jira 승인(Production) + Slack(Team). 확인 게이트 모호
[7] Testability                   FAIL — 책임 분리 없음. 실패 원인 구분 불가

FAIL이 나오면 고치고 다시 돌립니다. 이 루프를 통과한 최종 문서는 이렇게 생겼습니다.

## execute@approve-db-changes

**Input**: Slack 메시지 링크 (https://team.slack.com/archives/... 형식)
**Output**: Jira 티켓 승인 완료 메시지 또는 에러

### 프로세스
1. Slack 링크에서 SQL 쿼리 본문을 추출한다
2. SQL 문법 검증을 수행한다 (구문 오류 여부만 판단)
3. Jira 티켓 상태를 조회한다
4. 사용자에게 승인 내역을 보여주고 Y/N 확인을 받는다
5. Y인 경우에만 Jira 티켓을 승인 처리한다

### 불변 조건
- 실행 순서: SQL 추출 → 문법 검증 → Jira 조회 → 사용자 확인 → 승인. 변경 불가.
- SQL 검증이 실패하면 이후 단계를 진행하지 않는다.
- 사용자가 Y를 입력하기 전까지 Jira API를 호출하지 않는다.

### 금지 사항
- SQL 쿼리를 수정하거나 대안을 제안하지 마라
- SQL 성능 분석, 실행 계획 확인, 인덱스 제안을 하지 마라
- Jira 티켓 상태를 검증 없이 변경하지 마라
- 승인 사유를 AI가 자체 작성하지 마라
- 사용자 확인 없이 승인을 진행하지 마라
- 에러 발생 시 재시도하지 마라

### 엣지 케이스
- Slack 링크가 아닌 입력 → "Slack 메시지 링크를 입력해주세요" 반환
- Slack 링크이나 SQL이 없는 메시지 → "SQL 쿼리를 찾을 수 없습니다" 반환
- Jira 티켓이 이미 승인된 상태 → "이미 처리된 티켓입니다" 반환
- Jira 티켓이 존재하지 않음 → "티켓을 찾을 수 없습니다" 반환
- API 응답 실패 → 재시도 없이 에러 메시지 반환
- 사용자가 N 입력 → "승인이 취소되었습니다" 반환

### Requires
- JIRA_API_TOKEN
- SLACK_USER_TOKEN

처음 3줄이었던 문서가 이렇게 됐습니다. 입력 타입, 출력 타입, 실행 순서 강제, 금지 항목 6개, 분기 조건, 에러 핸들링, 의존성 선언까지 담겼습니다. 한국어를 걷어내고 기호로 바꾸면 코드나 다름없습니다.

근데 이러면 LLM 쓸 이유가 있나

해석의 여지가 남아 있다는 건 문제이기도 하지만, 동시에 LLM이 강력한 이유이기도 합니다. 빈 곳을 자기 판단으로 채웁니다. "SQL 검증을 수행한다"라고만 써도 맥락을 보고 알아서 문법 검사를 하기도 하고 성능 분석을 하기도 합니다. 코드로 이걸 하려면 분기를 전부 짜야 합니다.

제가 한 건 이 유연함을 체계적으로 없애는 작업이었습니다. 극단적으로 밀어붙이면 완벽하게 제약된 프롬프트는 그냥 코드입니다. 그럴 거면 LLM을 쓸 이유가 없습니다.

프레임워크를 만들면서 찾은 나름의 기준은, 되돌릴 수 없는 행위일수록 닫아야 한다는 것이었습니다. DB를 바꾸는 SKILL, Jira 티켓을 승인하는 SKILL, 프로덕션 배포처럼 되돌리기 어려운 작업에서는 AI의 재량이 리스크가 됩니다. 반대로 코드 리뷰, 리서치, 초안 작성 같은 건 열어둘 가치가 있습니다. 제가 못 본 관점을 채워주는 게 LLM의 가치이기 때문입니다.

결국 기준은 하나입니다. AI가 틀렸을 때 그 비용을 감당할 수 있는가. 감당할 수 있으면 열어두고, 없으면 닫습니다.

테스트에 대해서

닫아야 하는 이유가 하나 더 있습니다. 테스트입니다.

코드 테스트는 단순합니다. 입력을 넣고 출력을 비교합니다. assertEquals(expected, actual). 같은 입력이면 같은 출력이 나온다는 전제가 있어야 성립합니다.

에이전트는 이 전제가 통하지 않습니다. 같은 SKILL, 같은 입력인데도 대화 맥락에 따라 결과가 달라집니다. 어제 통과한 테스트가 오늘 깨지고, 모델 버전이 바뀌면 전부 깨지기도 합니다.

고민 끝에 내린 결론은, 출력 값을 검증하는 대신 행위의 경계를 검증하자는 것이었습니다. "Slack 메시지 보내기 전에 사용자 확인을 거쳤는가", "SQL 검증 단계를 건너뛰지 않았는가", "입력이 비었을 때 임의의 값을 넣지 않았는가" 같은 질문입니다.

여기서도 비슷한 역전이 나타납니다. 코드 테스트가 확인하는 건 "맞는 걸 했는가"이고, 에이전트 테스트가 확인해야 하는 건 "틀린 걸 안 했는가"에 가깝습니다. 단위 테스트보다는 보안 테스트에 가까운 사고방식입니다.

물론 음성 검증만으로는 "아무것도 안 하는 에이전트"가 만점을 받는 문제가 생깁니다. 금지 사항을 전부 지켰는가라고 물으면, 네, 아무것도 안 했으니까요, 라는 답이 나오는 셈입니다. 양성 검증도 당연히 필요합니다. 다만 우선순위의 문제라고 봅니다. DB를 날리지 않는 게 먼저고, 최적의 쿼리를 짜는 건 그다음입니다. 안전이 확보된 위에서 성능을 논해야 순서가 맞습니다.

그리고 경계를 검증하려면 경계를 먼저 정의해야 합니다. 금지 항목, 불변 조건, 확인 게이트 같은 것들입니다. 이게 없으면 테스트 자체가 성립하지 않습니다. 결국 에이전트를 테스트 가능하게 만드는 작업은 해석의 여지를 줄이는 작업과 같습니다. 테스트하려면 제약이 필요하고, 제약을 걸면 유연함은 줄어듭니다. 그렇다고 테스트 없이 프로덕션에 올릴 수는 없습니다.

정리

전통적인 코드 작성이 "무엇을 하라"를 쓰는 일이었다면, 프롬프트를 쓰는 일은 "무엇을 하지 마라"를 정의하는 쪽에 가깝습니다. 자연어에 제약을 충분히 걸면 코드와 구별되지 않습니다. 그 제약을 정의하고 검증하고 강화하는 과정은 문법을 만드는 과정과 다르지 않습니다.

다만 전부 닫을 필요는 없습니다. 전부 닫으면 LLM을 쓸 이유가 없고, 전부 열면 의도를 벗어납니다. 닫는 법을 아는 것 못지않게, 열어두는 법을 아는 것도 중요하다고 생각합니다.
